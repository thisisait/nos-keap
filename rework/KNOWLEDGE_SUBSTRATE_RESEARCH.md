# Knowledge substrates for LLMs — research digest & KEAP overlay proposal

*2026-07-11 · deep-research run (22 sources fetched, 109 claims extracted, 25 adversarially
verified: 24 confirmed 3-0, 1 refuted 0-3). Findings below carry only verified claims;
interpretation for KEAP is marked as such.*

## TL;DR

The field converged on **two complementary substrate styles**, and KEAP already has one of
them built:

1. **File-native "LLM-wiki"** — Karpathy's pattern (April 2026 gist), formalized by Google
   into the **Open Knowledge Format** (OKF v0.1, 2026-06-12): plain markdown + YAML
   frontmatter, graph emerges from ordinary markdown links. No DB, no RDF, no SDK.
2. **Hybrid vector+graph retrieval** — empirically effective **only in the right
   configuration**: entity/relation edges enriched with natural-language descriptions win;
   edges built from *vector similarity* actively hurt (worse than a flat index). Graph
   advantage grows with corpus scale.

KEAP's libSQL substrate (taxonomy graph + F32_BLOB vectors + FTS5) is the second style;
the proposal below adds an OKF-aligned **knowledge-object overlay** as the first — which is
also the answer to "capturing richer data than notes".

## Verified findings

### 1. Open Knowledge Format (Google Cloud, v0.1 Draft, 2026-06-12)

- Open, vendor-neutral spec that **formalizes Karpathy's LLM-wiki pattern**; explicitly
  designed to be *authored by people, generated by agents, consumed by both*.
  ([blog](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/),
  [SPEC.md](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md))
- Deliberately minimal: directory of markdown files with YAML frontmatter. **Only required
  field: `type`.** Standard fields: `title`, `description`, `resource` (URL to the live
  thing), `tags`, `timestamp`. No schema registry, no central authority, no required tooling.
- **Graph is emergent from markdown links** (SPEC §5.3): a link asserts a directed,
  *untyped* relationship; the kind of relationship lives in surrounding prose. Consumers
  derive graph views.
- Ecosystem: [openknowledge CLI](https://github.com/openknowledge-sh/openknowledge)
  (v0.5.0, 2026-07-07) — git-native knowledge bases, converts repos/folders/websites into
  OKF bundles, context assembly via **deterministic BM25 under a token budget** (no vectors).
- *Caveat: v0.1 Draft less than a month old; third-party adoption unknown; one independent
  implementer so far.*

### 2. Karpathy — LLM-wiki & cognitive core

- **LLM-wiki** ([gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)):
  instead of query-time RAG, the LLM *incrementally builds and maintains* a persistent wiki
  of interlinked markdown. Three layers: immutable raw sources → LLM-generated wiki
  (summaries, entity pages, cross-refs) → schema doc (CLAUDE.md/AGENTS.md). Three
  operations: **ingest** (batch-update many pages per new source), **query** (synthesize +
  write valuable answers back as new pages), **lint** (periodic check for contradictions,
  stale claims, orphan pages).
- **Cognitive core** ([tweet](https://x.com/karpathy/status/1938626382248149433)): a
  few-billion-param always-on model that *sacrifices encyclopedic knowledge for capability*,
  aggressively tool-using; facts externalize into on-demand lookup. *This is exactly the
  role cortex/KEAP plays for nOS agents — the external knowledge substrate.*

### 3. Do graphs actually help? (empirics)

- **"Does Memory Need Graphs?"** (arXiv 2601.01280, PKU, 01/2026, LongMemEval):
  - Similarity-built edges (SimGraph) **degrade** retrieval below a flat vector index
    (R@5 0.842 vs 0.905).
  - Entity-relation graph + natural-language entity descriptions (DescGraph) is **best**
    (R@5 0.933).
  - Graph advantage **grows with history scale** (LME-M: 0.828 graph vs 0.723 flat).
- **SAP "Towards Practical GraphRAG"** (arXiv 2507.03226): dependency-parsing KG
  construction reaches ~94 % of LLM-extraction quality at a fraction of cost; hybrid
  retrieval = dense similarity ⊕ one-hop graph traversal fused by **Reciprocal Rank Fusion
  (RRF, k=60)** + re-rank → up to +15 % over vanilla vector retrieval. *(Single internal
  benchmark, LLM-as-judge — treat directionally.)*
- **HybridRAG** (arXiv 2408.04948, BlackRock/NVIDIA): naive concatenation of vector+graph
  context improves answer relevance (0.96 vs 0.91/0.89) **but** the claim "hybrid beats
  both" was **refuted 0-3** in verification — its context precision (0.79) is far worse
  than GraphRAG's (0.96). Rank-level fusion (RRF) is the principled approach, not
  concatenation.
- **Graphiti / Zep** ([repo](https://github.com/getzep/graphiti), arXiv 2501.13956):
  bi-temporal fact model — superseded facts are **invalidated, never deleted** (validity
  windows on edges); query "what is true now" or "what was true at time T". Directly
  relevant to the *Preserver* half of KEAP's name.

### Design lessons for KEAP (interpretation)

| Lesson | KEAP status |
|---|---|
| Don't *store* similarity edges; entity+description edges only | ✅ already right — explorer draws similarity as an ephemeral view (star field), taxonomy is the stored graph. Keep it that way. |
| Hybrid retrieval should be RRF fusion, not fallback/concat | ◻ today `/agent/v1/search/semantic` falls back FTS→vector; upgrade to FTS5(BM25) ⊕ vector ⊕ one-hop taxonomy neighbors via RRF |
| Descriptions are load-bearing (DescGraph) | ✅ curated `description` is already part of canonical embed text |
| Graph pays off with scale | ✅ 790 nodes + growing captures/objects — we're on the right side |
| Preserve by invalidation, not deletion | ◻ backlog: `valid_from`/`invalidated_at` on knowledge objects (bi-temporal lite) |
| Lint operation (contradictions, orphans, stale) | ◻ future Pulse job candidate (`keap-lint`), mirrors keap-embed-sync |

## Proposal: knowledge-object overlay ("richer data than notes")

Owner requirement (2026-07-11): *a system/overlay for capturing more complex data than
notes.* Today KEAP captures only two shapes — todos (free text) and page captures
(companion metadata). The research points at an obvious, standards-aligned third shape:

### `knowledge_objects` — OKF documents in the database

One row = one OKF concept document. Same fields, same semantics:

```
knowledge_objects
  id            TEXT PK            (slug or uuid)
  user_id       TEXT               (human uid or 'agent:<name>')
  visibility    TEXT DEFAULT 'private'   -- sharing-ready like everything else
  type          TEXT NOT NULL      -- OKF's one required field: 'note', 'recipe',
                                   -- 'dataset', 'contact', 'howto', 'spec', … open-ended
  title         TEXT
  description   TEXT
  resource      TEXT               -- our requiredData grammar: 'kiwix:…', 'nextcloud:…', URL
  tags          TEXT (JSON array)
  frontmatter   TEXT (JSON)        -- arbitrary producer-defined structured fields
  body          TEXT               -- markdown; [[links]] / md links = graph edges
  links         TEXT (JSON array)  -- extracted refs: taxonomy node ids, object ids, URLs
  content_hash  TEXT               -- feeds the existing embed-sync pending diff
  created_at / updated_at
```

Why this shape wins:

1. **The overlay is typed but schema-free** — `type` + freeform `frontmatter` means a
   recipe, a hardware inventory row, a person, a survival checklist all fit without
   migrations. Exactly OKF's "minimally opinionated" principle.
2. **Graph for free** — `links` extracted from body (markdown links + `[[wiki]]` syntax to
   taxonomy nodes/objects) become untyped directed edges; the explorer renders objects as a
   new source kind alongside taxonomy/captures. No similarity edges stored (lesson #1).
3. **Vectors for free** — the embeddings table is already generic `(kind, ref_id)`; add
   `kind='object'`, extend the canonical-text assembler, and the existing keap-embed-sync
   Pulse job picks objects up with zero nOS-side changes.
4. **Agents can preserve, not just capture** — `POST /agent/v1/objects` (rw scope): an
   AgentKit agent that learns something durable writes an object, Karpathy's "query writes
   valuable answers back as new pages".
5. **OKF export/import = the sharing unit** — a bundle is just these rows serialized to
   `type/…/slug.md` with frontmatter. Phase S sharing becomes "export bundle, import
   bundle", interoperable with Google's tooling and openknowledge CLI. `resource` even
   round-trips our service refs.

### UI: capture beyond notes

- Admin/homepage "New object" form: type picker (recent types as suggestions), title,
  markdown body with `[[` autocomplete against taxonomy + objects, optional frontmatter
  key-value editor. The companion panel can offer "capture as object" for structured grabs.
- Explorer side panel gains an `objects` source kind; detail drawer renders the markdown
  body and its outbound links.

### Sequencing (proposal, needs owner go)

1. **O1** — table + CRUD `/api/objects` + agent `POST/GET /agent/v1/objects` + embed kind.
2. **O2** — link extraction → explorer integration (new source kind, edges).
3. **O3** — OKF bundle export (zip of markdown) / import with dedup by id+hash.
4. **O4** — RRF hybrid search upgrade (applies to whole corpus, not just objects).
5. Backlog — bi-temporal invalidation; `keap-lint` Pulse job.

## Open questions from the research

- OKF adoption beyond Google is unproven — but the format is so thin that betting on it
  costs almost nothing; worst case we own a clean markdown+frontmatter export.
- Edge typing: OKF leaves relations untyped (prose carries meaning) while memory research
  favors entity-relation+description graphs. KEAP's answer: taxonomy = typed hierarchy,
  object links = untyped, descriptions everywhere — a defensible middle point.
- Bi-temporal facts on a file-native substrate is unsolved in the wild (Graphiti needs a
  graph DB). Our libSQL rows + `invalidated_at` may be the pragmatic hybrid.
